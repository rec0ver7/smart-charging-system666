## 二、 组员精细化开发职责与接口指导

### 💾 1. 数据模型与计费模块 —— 【组员 C 优先开工】

* **负责文件**：`charging_system/models.py`、`charging_system/services/bill_service.py`、`charging_system/admin.py`
* **具体落实动作**：
* **`models.py`（数据模型层）**：使用 Django ORM 优先定义三个核心实体类：
1. `Car`（车辆类）：包含车辆ID、电池总容量、请求充电量、快慢充电模式、当前所处状态（等候区排队/充电区排队/充电中/已充满退出）、当前分配的充电桩ID、等候时间戳。
2. `ChargePile`（充电桩类）：包含充电桩ID、充电桩类型（快充/慢充）、运行状态（工作/空闲/故障）、当前排队车辆数、历史累计充电量、历史累计充电次数。
3. `Bill`（账单类）：包含账单流水号、车辆ID、充电桩ID、实际充电量、充电持续时长、充电启动时间、充电结束时间、分时充电费、分时服务费、总费用。


* **`admin.py`（管理后台注册）**：将上述三个类注册至 Django Admin，确保随堂验收演示时，能够直接通过后台修改充电桩状态（模拟突发故障场景）。
* **`bill_service.py`（业务逻辑层）**：
* 编写 `Request_Bill()` 与 `Request_DetailedList()` 函数。
* 实现**分时计费规则引擎算法**：严格根据 PPT 规定的“尖峰平谷”多时段不同电价及固定服务费率公式，结合车辆充电的绝对起止时间进行分段积分计算，输出费用详单。
* **加入时间缩放系数**（例如真实运行1秒 = 模拟时间1分钟），确保在数分钟的演示验收时间内，费用可以发生跨时段动态变化。





### 🧠 2. 中央调度算法模块 —— 【组长、组员 B、组员 D 协同】

* **负责文件**：`charging_system/services/dispatch_service.py`
* **具体落实动作**：
* **组长（正常调度逻辑与核心算法）**：
* 编写 `E_chargingRequest()` 核心叫号分配接口。
* 编写**优先级调度算法**：在等候区车辆准备调入充电区时，综合评估快慢充模式及车辆排队先后顺序，计算并分配最合理的充电桩队列。
* 编写**时间片轮询算法**：处理多辆车在同一桩排队时的轮转控制，编写切块时间逻辑。


* **组员 D（故障备用时序调度）**：
* 编写**时间顺序调度算法**：在特定或故障模式下，确保等候区及充电桩后方队列严格基于时间戳的公平排队与桩位分配逻辑。


* **组员 B（故障重调度恢复逻辑）**：
* 编写 `handle_pile_fault(fault_pile_id)` 函数。
* 当充电桩在工作状态中发生突发故障时，该函数须立即锁定受影响的充电桩，提取该桩正在充电及后方排队的全部车辆，执行**再调度策略**，将受影响车辆无缝重归等候区或其他正常工作的充电桩后方队列。





### 🚗 3. 车辆指令与状态维护模块 —— 【组员 A、组员 B 协同】

* **负责文件**：`charging_system/services/car_service.py`
* **具体落实动作**：
* **组员 A（车主端控制指令与查询）**：
* 编写 `Modify_Amount(car_id, new_amount)`：支持用户在中途修改目标充电量。
* 编写 `Modify_Mode(car_id, new_mode)`：支持用户在中途切换快慢充模式，并触发调度机制重新计算其排队顺序。
* 编写 `Query_Car_State(car_id)`：向用户返回其当前车辆的精准状态及前方的排队等待车辆数。


* **组员 B（充电状态物理机控制）**：
* 编写 `Start_Charging(car_id)`：控制车辆正式进入充电区并灌入流速电量，变更车辆及桩的相关状态。
* 编写 `End_Charging(car_id)`：实现充满自动停止或用户中途主动取消充电的停止逻辑，完成电量锁定并触发计费结算。
* 编写 `Query_Charging_State(car_id)`：实时计算并返回该车当前已充电度数、当前已产生的瞬时费用等数据。





### ⚡ 4. 充电桩管理与监控模块 —— 【组员 D、组员 E 协同】

* **负责文件**：`charging_system/services/pile_service.py`
* **具体落实动作**：
* **组员 D（管理员基础控制指令）**：
* 编写 `powerOn(pile_id)` 与 `powerOff(pile_id)`：实现管理员对指定充电桩的硬开机与硬关机控制。
* 编写 `setParameters()`：支持管理员在线调整和覆盖全局的时段电价及服务费比例参数。
* 编写 `Start_ChargingPile(pile_id)`：激活初始化后的充电桩进入待机/工作状态机。


* **组员 E（管理员监控大屏数据引擎）**：
* 编写 `Query_PileState()`：收集并封装全部 5 个充电桩的全局运行指标（如累计电量、充电频次、当前状态机状态等），作为前端大屏图表的数据源。
* 编写 `Query_QueueState()`：获取各个充电桩后方专属排队队列（4个车位限制）中详细的车辆列表与流转数据。
* **系统串联职责**：作为架构搭建与技术报告主撰人，须在前端模拟界面或文档中详细说明：前端通过周期循环（`loop` 定时轮询）机制，每隔固定周期请求本文件内的查询接口，从而在不影响后端性能的前提下实现管理员监控大屏的动态刷新。

## 三、 全员协作开发工作流纪律

1. **并行开发依赖机制**：组员 C 须在本日内优先完成 `models.py` 的编写，并在本地执行：
```bash
python manage.py makemigrations
python manage.py migrate

```


随后将生成的日志提交 Pull Request 合并至云端主分支。此时，`migrations/` 目录下将自动生成变动日志文件。其余组员在接收到组长通知后，先执行 `git pull origin main` 同步本地数据库地基，随后即可在各自负责的 `services/` 子文件中独立编写算法，互不干扰。
2. **算法独立调试**：在前端框架或路由未就绪前，全员可直接将测试样本注入应用根目录下的 `tests.py`。通过编写本地 Mock 对象运行控制台测试，提前验证各复杂调度与计费算法的准确性。
3. **接口视图（Views）组装**：业务逻辑层（Services）函数全部通过测试后，全员须协同在应用根目录下的 `views.py` 中，将各自编写的 Service 函数进行 RESTful API 转换包装，为前端提供统一的标准化 JSON 数据接口。
